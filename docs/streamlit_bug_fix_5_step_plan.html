2Problems — ช่องว่างที่ขวางอยู่

P1 — DataFrame ผสม index ชนิดข้อความ (+4…0) กับ pandas Timestamp แล้วส่งตรงเข้า st.dataframe; PyArrow ต้องสร้างคอลัมน์ ↓ index ชนิดเดียวจึงแปลงไม่ได้severity 5/5 · gap 3

P2 — ยังมีการเรียก st.components.v1.iframe แบบเก่ากระจายอย่างน้อย 15 จุดใน 3 ไฟล์ ทำให้เกิด deprecation warning และมีความเสี่ยงพังหลัง API ถูกถอดseverity 4/5 · gap 15

P3 — process Streamlit จบด้วย Segmentation fault ซึ่งเป็น native crash; traceback Python ปกติจับไม่ได้ และ dependency native สำคัญไม่ถูก pinseverity 5/5 · gap 1

P4 — ticker no-data/delisted ถูกเรียกซ้ำผ่าน Yahoo หลาย fallback/หลาย thread ทำให้เกิด error noise, เพิ่มโหลด และทำให้การหาต้นเหตุ segfault ยากขึ้นseverity 3/5 · gap 2

3Diagnosis — รากของปัญหา

RC1 — pd.concat รวม Index[str] ของ future rows กับ DatetimeIndex ของ history rowsw = 0.75

  1. ต้องการให้ตารางมีแถวอนาคต +4 ถึง 0 อยู่เหนือข้อมูลราคาเก่า
  2. จึงสร้าง future_df ด้วย index แบบ string แล้ว concat กับ history ที่ index เป็น Timestamp
  3. index ที่ผสมชนิดถูกตั้งชื่อ ↓ index และ Streamlit/PyArrow พยายาม serialize เป็นคอลัมน์เดียว
  4. Arrow เลือก representation แบบข้อความ/bytes แต่พบ Timestamp จึงโยน Expected bytes, got Timestamp

หลักฐาน: ยืนยันจาก 📈_Monitor.py, pages/Calculator.py และ pages/3_🕹_Test_F(Z): future_index เป็น string, history เป็น DatetimeIndex, concat แล้วตั้ง combined_df.index.name='↓ index'

RC2 — ขั้นตอน sanitize ก่อน st.dataframe แปลงเฉพาะ data columns แต่ไม่ normalize DataFrame indexw = 0.25

  1. โค้ดวน astype(str) เฉพาะคอลัมน์ dtype object
  2. ↓ index ยังอยู่ใน Index ไม่ได้อยู่ใน df.columns
  3. จึงยังคงมี string และ Timestamp ผสมกันตอนส่งเข้า st.dataframe

หลักฐาน: ยืนยันใน expander Show Raw Data Action ของ 📈_Monitor.py และ st.dataframe(asset_df) ของ Calculator

RC3 — iframe legacy ถูกเรียกตรงหลายจุดใน CF_Graphw = 0.65

  1. หน้าเดิมฝังหลาย artifact โดยเรียก st.components.v1.iframe ทุกครั้ง
  2. ไม่มี helper กลางสำหรับ API ใหม่
  3. เมื่อ Streamlit เปลี่ยน API ทุก call จึงเตือนซ้ำ

หลักฐาน: pages/0_🔥_CF_Graph.py มี 11 source occurrences

RC4 — helper iframe ใน Principle_F(X) และ Book_F(X) ยังห่อ API เก่า และ Book ใช้ @st.cache_data ครอบฟังก์ชัน UIw = 0.35

  1. สร้าง helper เพื่อเรียกซ้ำสะดวก
  2. helper ไม่ได้ถูกอัปเดตเมื่อมี st.iframe
  3. การ cache ฟังก์ชันที่วาด UI ไม่จำเป็นและเพิ่ม state/behavior ที่คาดเดายาก

หลักฐาน: pages/8_📕_Principle_F(X).py และ pages/9_📑_Book_F(X).py

RC5 — requirements.txt เปิดช่วงเวอร์ชันกว้างและไม่ pin numpy/pandas/streamlit/yfinance/curl_cffi/pyarroww = 0.45

  1. ทุก deploy สามารถดึง wheel รุ่นใหม่ต่างจากรอบก่อน
  2. ไลบรารีเหล่านี้มี native extension/ABI ที่ process สามารถ crash ได้โดยไม่มี Python exception
  3. จึง reproduce และ rollback ยาก

หลักฐาน: ยืนยันว่า requirements.txt ไม่ pin; ความเชื่อมโยงกับ segfault เป็นสมมติฐานที่ต้องทดสอบด้วย lockfile matrix

RC6 — Yahoo calls ทำงานพร้อมกันสูงสุด 8 threads และแต่ละ ticker มีหลาย fallback (fast_info → info → history)w = 0.35

  1. ต้องการให้ราคาหลาย ticker โหลดเร็ว
  2. จึงใช้ ThreadPoolExecutor(max_workers=min(8,len(tickers)))
  3. network/native stack ของ yfinance/curl_cffi ถูกเรียกพร้อมกันจำนวนมาก
  4. เมื่อมี ticker no-data จะเกิด fallback/retry เพิ่ม ทำให้ crash surface ใหญ่ขึ้น

หลักฐาน: ยืนยัน concurrency/fallback จาก 📈_Monitor.py; เป็น probable cause ไม่ใช่ข้อสรุปจนกว่าจะ A/B test

RC7 — ไม่มี faulthandler, minimal reproduction และ cold-start stress gate สำหรับ native crashw = 0.20

  1. log ปัจจุบันจบที่ shell รายงาน Segmentation fault
  2. ไม่มี native-adjacent Python stack dump ก่อน process ตาย
  3. จึงไม่รู้ว่า crash เกิดระหว่าง Arrow, curl_cffi, numpy หรือโมดูลอื่น

หลักฐาน: ไม่พบ crash instrumentation ในไฟล์หลักที่ตรวจ; ต้องยืนยันเพิ่มเติมทั้ง repo

RC8 — ไม่มี ticker preflight/health status ที่แยก active, unavailable และ delisted ก่อนโหลดหน้าหลักw = 0.70

  1. ticker มาจาก config หลายไฟล์
  2. ทุก ticker ถูกส่งเข้า Yahoo เหมือนกัน
  3. symbol ที่ไม่มีข้อมูลจึงถูกลองซ้ำและส่ง error log ทุก rerun

หลักฐาน: log ระบุ FFWM/APLS no price data; repo มี ticker เหล่านี้ใน config/หน้าอื่น

RC9 — ไม่มี negative cache/circuit breaker สำหรับผล no-dataw = 0.30

  1. cache ปัจจุบันเน้นค่าราคาที่สำเร็จ
  2. ผลล้มเหลว 0.0 ไม่ได้มีสถานะเหตุผลและอายุ cache ที่ชัดเจน
  3. ผู้ใช้ rerun แล้ว request เดิมเกิดซ้ำ

หลักฐาน: get_cached_price คืน 0.0 หลัง fallback แต่ไม่ได้บันทึก unavailable reason หรือ cooldown
